## Done

- **2026-07-16** — Fixed broken screen routing in `companion-app`. Four screens (`SetupMissions`, `PlayMissions`, `AdventuresSetup`, `AdventuresPlay`) were built but unreachable because `App.tsx` had no route for their `screen` values (`setup`, `play`, `advSetup`, `advPlay`), and a couple of `setScreen('status')` calls didn't match the `'status-effects'` branch. Added a `Screen` union type in `store.tsx` so `AppState.screen`/`setScreen` catch mismatches at compile time, added the missing routes in `App.tsx`, and fixed the mismatched calls in `Hub.tsx` / `MissionsDashboard.tsx`. Verified all four screens reachable via their real navigation paths in the browser. Merged `claude/code-structure-review-a6cf04` → `main` (fast-forward, `92a03e8`).
- **2026-07-16** — Wired up placeholder/mocked data across the app. Turned out `constants.ts` already had all six real data sets extracted from the design file (`RULES_DATA`, `STATUS_DEFS`, `PERK_DEFS`, `HERO_LIST`, `CRAWL_PHASES`, `BOSS_PHASES`) but almost nothing imported them:
  - `StatusEffects.tsx` and the status chips in `Trackers.tsx`/`PvP.tsx` now use the real 22-entry `STATUS_DEFS` (added `src/lib/status.ts` as the shared source instead of duplicating a mock list in three places) — stack limits are now per-status instead of a hardcoded 3.
  - `Rules.tsx` now uses the real 73-entry `RULES_DATA` (was a 4-item placeholder) and renders `callout`/`bullets` it was previously dropping.
  - `Perks.tsx` was a pure UI mockup with no data at all — now uses real `PERK_DEFS`, split into Combat/Momentum sections, and the per-column grid was reworked from a single on/off toggle to a filled-circle-count model (matching `cols: [2,0,0,0,0]` in the real data). Also wired to the store's existing-but-unused `perkPlayers`/`activePerkPlayer`/`sessionPP` fields so progress now persists per hero instead of resetting on navigation.
  - `AdventuresPlay.tsx` phase rail now shows real `CRAWL_PHASES`/`BOSS_PHASES` content (subtitle/body/callout/bullets) instead of a placeholder paragraph that literally said "this is where body copy would be injected."
  - `Adventures.tsx` scoring sheet: Win/Loss buttons and the Salves input were dead (no `onClick`/uncontrolled). Wired both to state, plus a live Campaign Total and the Salves carry-forward chain (win carries remaining Salves, loss adds +3, per the actual rule text). Deliberately did *not* invent the difficulty-card point values for scenario scoring since they're not in `constants.ts` or the rulebook data — Score stays a manual per-row entry (auto-seeded to −10 on a Loss, since that number is an actual stated rule) rather than a fabricated formula.
  - Verified all of the above end-to-end in the browser (status grid, rules browsing incl. See Also links, perk unlock toggling, adventure phase content, and scoring math).
- **2026-07-16** — Missions & Adventures rework, using the real rulebook PDFs the user provided (`G:\My Drive\Gaming\Dice Throne\*.pdf`, extracted to text via `pdftotext -layout` since `pdftoppm` isn't installed in this environment — text extraction was clean enough to grep/verify against).
  - **Found and fixed a real content bug**: `boss-step`, `pvp-boss-vs-many`, and `pvp-boss-vs-boss` in `RULES_DATA` were tagged `cat:'missions'`/`cat:'pvp'` but their content (Action decks, Chaos dice, King's Hand tokens, "teammate operates the Boss") is 100% Adventures-only — confirmed by grepping the base 498-page PVP rulebook (zero hits for "boss" anywhere in it) and finding the real source in the Adventures rulebook's "boss vs many mode"/"boss vs boss mode" section. Recategorized all three to `cat:'adventures'`. Also fixed `targeting-roll` (was a `cat:'core'` entry that incorrectly implied Missions uses a Chaos-die targeting roll — Missions has no Chaos dice at all; Multi-Strike just hits everyone in range) and `boost-tiles`' now-stale `seeAlso` link. **Important**: these fixes had to go into `Dice Throne Companion - Hi-Fi.dc.html` (the source `extract.cjs` regenerates `constants.ts` from) — editing `constants.ts` directly would've been silently overwritten on the next `npm run dev`/`build`.
  - Also fixed `SetupMissions.tsx`'s starting-HP-by-hero-count ledger, which showed placeholder numbers (50/40/35/30) — real rulebook values are 45/30/25/20 for 1–4 heroes (pg. 4, "Choosing a Mission"). Same file's "Enemy Dice Pool" label was showing fabricated "3x Minion, 1x Henchman" (Missions doesn't have Minions — that's Adventures terminology) — now shows the real formula (2 Red + 1-per-hero Black), also fixed in `Trackers.tsx` and `PlayMissions.tsx`.
  - Added `src/data/missionPhases.ts` (hand-authored, not machine-extracted — there's no Missions phase content in the design source to extract) with accurate `MISSION_SIDE1_PHASES`/`MISSION_SIDE2_PHASES`, covering the 8 Round Phases for both sides of a Mission. The key rule difference the user specifically asked about: on Side 1 (Henchmen), *every* Henchman type whose roll objective matches a single dice roll activates, resolved left-to-right — not just one. On Side 2 (Boss), only the *first* matching Offensive Ability activates, plus Boss Defense (auto-activates on defendable Attacks, no roll) and the face-down Boost Tile/CRIT mechanic. Wired into `PlayMissions.tsx` with a Side 1/Side 2 switcher, replacing the old hardcoded 8-phase name list and mocked dice-tile display.
  - Added `bossHp`/`bossMaxHp` to the store and a shared `BossHpTracker.tsx` component, used in `PlayMissions.tsx` (Side 2 only), `Trackers.tsx`, and `AdventuresPlay.tsx` (Boss mode) — per the user's clarification that Adventures has its own separate Boss Battles from Missions Boss Fights.
  - Removed per-hero HP tracking from `Trackers.tsx` entirely (players track their own HP on their Health Dial now) — kept Momentum and Status Effects tracking. Removed the now-fully-dead `heroHp` field from `AppState`/`store.tsx` since nothing read it anymore after this change.
  - Verified all of the above end-to-end in the browser: Side 1 vs Side 2 phase content and the Enemy Dice Pool math, Boss HP stepper persisting to state, Trackers screen with no per-hero HP UI, SetupMissions showing the corrected 45/30/25/20 HP table, and the Rules screen category counts shifting correctly after the recategorization (Missions 20→19, PVP 9→7, Adventures 16→19).
